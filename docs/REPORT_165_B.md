## Section 1 — Overview

Session 165-B resolves a model-vs-migration state drift on
`UserProfile.avatar_url` that was surfaced on 2026-04-21 when
Session 165-A's newly-activated Heroku release phase ran its
first `manage.py migrate`. Django reported the `prompts` app had
changes not yet reflected in a migration. Local diagnosis via
`makemigrations prompts --dry-run` confirmed exactly one drift:
`~ Alter field avatar_url on userprofile`.

Migration `0086_alter_userprofile_avatar_url.py` was generated
via `python manage.py makemigrations prompts` (a scoped exception
granted by this spec — CC still did NOT run `manage.py migrate`).
`sqlmigrate prompts 0086` confirmed the SQL is a pure no-op at
the PostgreSQL level (`BEGIN; -- (no-op) COMMIT;`). Post-gen
dry-run returned "No changes detected in app 'prompts'",
confirming 0086 captured all drift and no secondary drift lurks.

**Diagnosis discrepancy documented transparently:** The spec
originally framed the drift as `CharField → URLField`. Evidence
shows the drift is **help_text-only** — the field class has been
`URLField` throughout (introduced in migration 0084; preserved
via 0085's `RenameField` operation). The `help_text` string
changed in `prompts/models.py` during Session 163-B (commit
`de75e9c`) without a follow-up migration. The fix mechanism
(`makemigrations` → autogenerated `AlterField`) is identical for
either drift type, so implementation proceeded correctly despite
the spec's slightly-off diagnosis. Full evidence in Section 4.

Session 165 is now a two-spec session: 165-A (release phase)
committed earlier today as `4d874d4`; 165-B (this spec) extends
the same narrative arc — "structural deployment safety doing its
job on day one."

### Grep B — Migration 0085 field-class record

The spec EXPECTED Grep B to show `models.CharField(...)` for
`avatar_url`, which would have "confirmed" the spec's diagnosis.
Actual output:

```
$ grep -n "avatar_url" prompts/migrations/0085_drop_cloudinary_avatar_add_avatar_source.py
```

**Finding:** Migration 0085 does NOT re-declare the field class.
It uses `RenameField` to rename `b2_avatar_url` → `avatar_url`.
Django's `RenameField` operation preserves the field class +
metadata from the PREVIOUS migration's `AddField`. The field
class is inherited from 0084. The `help_text` on line 44 is
`avatar_source`'s (different field).

Reading migration 0084 directly (for full diagnosis):

```python
# prompts/migrations/0084_add_b2_avatar_url_to_userprofile.py
operations = [
    migrations.AddField(
        model_name='userprofile',
        name='b2_avatar_url',
        field=models.URLField(
            blank=True, default='',
            help_text='B2/Cloudflare CDN URL for avatar (replaces Cloudinary). '
                      'Populated by migrate_cloudinary_to_b2 management command. '
                      'Templates should check this first, fallback to Cloudinary avatar.',
            max_length=500
        ),
    ),

```

**Correct diagnosis:** 0084 recorded `URLField` with the ORIGINAL
help_text (from the b2_avatar_url era). Session 163-B (commit
`de75e9c`) updated the model's help_text in `prompts/models.py`
without generating a follow-up migration. 0085's RenameField
preserved 0084's field class (URLField) and metadata (old
help_text) while only renaming the field. Drift is **help_text-only**.

("CharField for avatar_url"), but the conditional STOP rules in

## Section 6 — Concerns

1. **Spec's diagnosis vs. evidence divergence.** The spec framed
   the drift as `CharField → URLField`. Evidence showed the drift
   is **help_text-only**. Migration 0085 uses `RenameField`,
   which preserves field class from 0084 (already `URLField`).
   The model's help_text was updated in `de75e9c` without a
   follow-up migration. The fix mechanism (autogenerated
   `AlterField`) is identical for either drift type, so
   implementation proceeded correctly after the `makemigrations
   --dry-run` pattern matched the spec's expected output
   exactly. The discrepancy is documented transparently in:
   - This Section 6
   - CHANGELOG Session 165-B subsection (Key decisions)
   - The corrected commit message (see Section 10)

   **Lesson for future specs:** Grep B expected Django's field
   class declaration to appear in the rename migration. For
   `RenameField` operations, field class is inherited from the
   previous migration's `AddField`, not re-declared. Future
   diagnostic greps should read the full `AddField` migration
   (or look at the migration-state frozen by `showmigrations
   --format=plan` + model inspection) rather than scanning the
   `RenameField` migration itself.

2. **Commit message correction.** The spec provided a fixed
   commit message claiming "migration 0085 state (recorded as
   CharField)". This is factually wrong per evidence. @architect-review
   flagged it as actionable before commit. Correcting the message
   to accurately describe help_text drift (see Section 10). The
   spec author was operating in good faith on a partially-
   incorrect diagnosis; the right response is to fix the message,
   not to propagate the inaccuracy for byte-for-byte conformance.

3. **sqlmigrate ran against SQLite locally.** DATABASE_URL is
   not set (per env.py safety policy), so sqlmigrate's no-op
   output is from SQLite, not PostgreSQL. For a help_text-only
   migration, the no-op property holds identically on both
   engines (help_text is never a database concern). Confirmed by
   @database-admin and @django-pro agents. No action needed; on
   the record in case a future reader wonders.

4. **First real release-phase migration on next deploy.** 165-A
   activated release phase but the first post-activation deploy
   had "No migrations to apply" (because migration 0085 was
   already applied manually during the 2026-04-19 incident
   recovery). The next deploy with 0086 will be the first actual
   release-phase auto-apply. Expected duration: 2–5 seconds.
   Not a concern — described in CHANGELOG and Section 9 for
   post-deploy verification.

5. **Migration file docstring convention.** @database-admin
   noted 0086 has no session-annotation docstring (0085 has a
   thorough one). Spec explicitly prohibits hand edits to the
   generated migration. Tradeoff: keep autogenerated output
   pristine (and trust sqlmigrate verification) vs. add a
   session annotation that could subtly alter the file's
   deconstructable form. Kept pristine per spec rule. Marked
   as deferred (Section 5) for future discussion.
