## Section 1 — Overview

GPT Image 2 was officially released by OpenAI on April 21, 2026 (5 days
before this session). It is the next-generation image model — adds
reasoning ("Thinking Mode"), 2K resolution, dramatically improved text
rendering across multiple scripts. Model ID: `gpt-image-2`. Available
via the standard OpenAI API.

This spec adds it to PromptFinder's bulk generator as a **BYOK-only**
selectable model, mirroring the existing `gpt-image-1-5-byok` pattern.
Platform-key access for gpt-image-2 is out of scope for this spec
(platform billing for OpenAI is a future Phase SUB consideration).

Five surfaces touched:
1. New `AI_GENERATORS['gpt-image-2-byok']` dict entry
2. New `AI_GENERATOR_CHOICES` tuple entry (added in Round 2 after agent
   review caught a silent-corruption gap)
3. New `GeneratorModel` registry row via migration 0090 (RunPython
   data migration, not schema change) AND mirrored in
   `seed_generator_models.py` for fresh-DB convergence
4. `OpenAIImageProvider` threads `model_name` through `__init__` →
   `self.model_name` → `client.images.edit/generate(model=...)`,
   replacing 2 hardcoded `'gpt-image-1.5'` literals
5. `tasks.py` `_run_bulk_generation_task` adds `elif job.provider ==
   'openai':` branch in `_provider_kwargs` setup, mirroring the existing
   Replicate branch
6. New choices migration 0091 (auto-generated by makemigrations after
   the AI_GENERATOR_CHOICES tuple change)

## Section 4 — Issues Encountered and Resolved

**Issue:** `AI_GENERATOR_CHOICES` in `prompts/models/constants.py` did
not include `('gpt-image-2-byok', 'GPT Image 2 (BYOK)')`.

**Root cause:** The spec specified adding the entry to AI_GENERATORS
(the dict in `prompts/constants.py`) and to seed_generator_models.py and
to the migration, but did not call out AI_GENERATOR_CHOICES (the tuple
in `prompts/models/constants.py`) — a separate file with similar name.
The Session 169-B canonical rule requires every GeneratorModel.slug to
also be in AI_GENERATOR_CHOICES because Prompt.ai_generator has
`choices=AI_GENERATOR_CHOICES`. Without the tuple entry, Django model
validation would silently reject `prompt.save()` for any GPT Image 2
publish — exactly the silent-corruption pattern that memory rule #13
was added to prevent.

**Fix applied:** Added `('gpt-image-2-byok', 'GPT Image 2 (BYOK)')` to
the AI_GENERATOR_CHOICES tuple in `prompts/models/constants.py:51`,
positioned between gpt-image-1-5-byok and flux-schnell. Ran
`makemigrations --name add_gpt_image_2_to_choices` to generate
migration `0091_add_gpt_image_2_to_choices.py` (AlterField on both
Prompt.ai_generator and DeletedPrompt.ai_generator — sibling-field
pattern matching Session 169-C). Both `python manage.py check` and
`python manage.py makemigrations --dry-run` confirm clean state
post-fix. Round 2 agent reviews (`@django-pro` 9.5/10, `@code-reviewer`
9.5/10) verified the fix closes the silent-corruption path.

**File:** `prompts/models/constants.py` line 51 + auto-generated
migration `prompts/migrations/0091_add_gpt_image_2_to_choices.py`.

## Section 5 — Remaining Issues

**Issue:** `IMAGE_COST_MAP` does not yet have per-model pricing.
gpt-image-2 jobs use gpt-image-1.5 prices for the cost-tracking
display.

**Recommended fix:** Restructure `IMAGE_COST_MAP` to be model-keyed
(`{model: {quality: {size: price}}}`) and update `get_image_cost()`
signature to accept a `model` parameter (Option B1 from spec section 5).
Update all callers in tasks.py and bulk_generator_views.py. Add
gpt-image-2 prices once OpenAI publishes per-quality/size data
(currently only token-based pricing is published).

**Priority:** P3 — gpt-image-2 is BYOK (OpenAI bills the user
directly); displayed cost is informational only, not a billing line.
The ~25-60% inaccuracy is bounded by the BYOK model and acceptable
until precise pricing is published.

**Reason not resolved:** Deferred — restructure adds risk to this
session's full suite gate; precise pricing data is not yet available
for gpt-image-2 (token-based only); the architectural change should be
done once with the right data, not iteratively. Captured as a P3
Deferred row in Spec D.
